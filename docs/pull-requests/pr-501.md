# [PR 501](https://github.com/vig-os/devcontainer/pull/501) fix: release rollback and retry exit codes

## Description

Fixes [#500](https://github.com/vig-os/devcontainer/issues/500): the CI `retry` helper now returns the failed command’s non-zero exit status after exhausting retries (so jobs fail correctly). Release workflow rollback no longer force-pushes; it creates a revert commit via the Git API so it works with branch protection on `release/*`. Includes BATS coverage for `retry` exit-code behavior. The branch also contains `revert: undo finalize release 0.3.2` so the release line can be corrected before re-finalizing.

This rollback logic updates the `release/$VERSION` ref via the Git Data API (create commit + PATCH ref). In this workflow, protected-branch git/ref operations are typically performed with the COMMIT_APP token (ruleset/branch-protection bypass), while RELEASE_APP is used for PR/issue operations; using RELEASE_APP here may still be blocked under the same protections that previously broke finalize/rollback. Consider generating a COMMIT_APP token in the rollback job and using it for the `gh api .../git/commits` and `.../git/refs/...` calls so rollback reliably works on protected `release/*` branches.

```diff
+# retry() — keep in sync with .github/actions/setup-env/action.yml (heredoc body)
+# and .github/workflows/sync-main-to-dev.yml (embedded copy).
+retry() {
+  local retries=3
+  local backoff=1
+  local max_backoff=60
+  local rc=1
+
+  while [ "$#" -gt 0 ]; do
+    case "$1" in
+      --retries)
+        retries="$2"
+        shift 2
+        ;;
+      --backoff)
+        backoff="$2"
+        shift 2
+        ;;
+      --max-backoff)
+        max_backoff="$2"
+        shift 2
+        ;;
+      --)
+        shift
+        break
+        ;;
+      *)
+        echo "ERROR: Unknown retry option '$1'"
+        return 2
+        ;;
+    esac
+  done
+
+  if [ "$#" -eq 0 ]; then
+    echo "ERROR: retry requires a command after '--'"
+    return 2
+  fi
+
+  local attempt=1
+  local current_backoff="$backoff"
+  while [ "$attempt" -le "$retries" ]; do
+    "$@" && return 0
+    rc=$?
+    if [ "$attempt" -lt "$retries" ]; then
+      local wait="$current_backoff"
+      if [ "$wait" -gt "$max_backoff" ]; then
+        wait="$max_backoff"
+      fi
+      echo "Retry $attempt/$retries failed (exit $rc), waiting ${wait}s..."
+      sleep "$wait"
+      current_backoff=$((current_backoff * 2))
+    fi
+    attempt=$((attempt + 1))
+  done
+
+  echo "ERROR: Command failed after $retries attempts: $*"
+  return "$rc"
+}
```

This introduces a third copy of the `retry()` implementation (action.yml heredoc, sync-main-to-dev inline helper, and this fixture). Even with the “keep in sync” note, this increases drift risk; consider making the workflows source the helper from a single tracked script (and have tests source that same script) so CI behavior and test coverage can’t diverge silently.
```suggestion
# Load the shared retry() helper from a single tracked script so tests exercise
# the same implementation used elsewhere in the repository.
_retry_helper_fixture_dir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
_retry_helper_repo_root="$(cd "${_retry_helper_fixture_dir}/../../.." && pwd)"
_retry_helper_shared_script="${_retry_helper_repo_root}/.github/actions/setup-env/retry_helper.bash"

if [ ! -f "${_retry_helper_shared_script}" ]; then
  echo "ERROR: Shared retry helper not found: ${_retry_helper_shared_script}" >&2
  return 1
fi

# shellcheck source=/dev/null
. "${_retry_helper_shared_script}"

if ! command -v retry >/dev/null 2>&1; then
  echo "ERROR: Shared retry helper did not define retry()" >&2
  return 1
fi
```
