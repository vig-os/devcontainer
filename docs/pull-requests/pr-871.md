# [PR 871](https://github.com/vig-os/devkit/pull/871) fix(ci): self-host an NVD feed mirror for vulnix scans

Closes #870.

## Problem

`nvd.nist.gov` is chronically throttled/unstable (`nix-community/vulnix#171`). Cold vulnix runs crawl the full NVD year-feed set for tens of minutes and fail with `ReadTimeoutError`/`IncompleteRead` â€” verified 2026-07-06, a cold run failed all 3 retries over **53 min**, blocking both the nightly gate and the release publish gate.

## Change

- **New `nvd-mirror-refresh.yml`** â€” the only job that talks to NVD. Downloads the 7 NVD 2.0 feed files (`nvdcve-2.0-<6 years>.json.gz` + `modified`) with resumable retry (`curl -C -`), validates them (`gzip -t`), and republishes them as assets on the `nvd-mirror` release. Never clobbers the mirror with a partial set.
- **`security-scan.yml` + `release.yml` `vulnix-gate`** now run `vulnix --mirror <nvd-mirror release>/`, fetching NVD data from GitHub's CDN instead of `nvd.nist.gov`. An NVD outage can no longer fail a scan or block a release.
- Docs (`docs/CONTAINER_SECURITY.md`) + CHANGELOG.

## Why a mirror (not just retries)

The mirror localizes the NVD dependency to one job: if the feeds ever change/retire, only the refresh job's source changes â€” vulnix keeps consuming identical files. vulnix has no NVD-API-migration plan and is on the current 2.0 feeds (no firm retirement date).

## Sequencing (important)

- A new workflow is only schedulable/dispatchable from the **default branch**. `main` is still the pre-cutover Debian world, so the scheduled refresh activates only once this rides the cutover onto the new `main`.
- Until then the `nvd-mirror` release is **bootstrapped out-of-band** (the 7 feeds uploaded manually), so this release's own `vulnix-gate` can use the mirror immediately.
