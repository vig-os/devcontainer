# [PR 744](https://github.com/vig-os/devkit/pull/744) fix(image): provide FHS loader + lib path for manylinux wheels

## Summary

The pure-Nix image (Nix-but-not-NixOS) could not run pre-compiled PyPI (manylinux) wheels installed at runtime. Two gaps:

1. No FHS dynamic loader `/lib64/ld-linux-x86-64.so.2` (the `PT_INTERP` every manylinux x86_64 wheel hardcodes) ‚Äî standalone PyPI tools (consumer pre-commit `ruff`/`typos`) aborted with `cannot execute: required file not found`.
2. The Nix C++/compression runtime (`libstdc++`/`libgcc_s`/`libz`) was off the loader path ‚Äî C extensions dlopened by the baked CPython (`numpy`/`scipy`, pre-commit `pymarkdown`'s `pyjson5`) failed with `ImportError: libstdc++.so.6`.

## Fix

- **Bootstrap layer**: symlink the Nix glibc loader to `/lib64/ld-linux-x86-64.so.2` (newer glibc runs old-glibc wheels via backward compatibility).
- **`config.Env`**: bake `LD_LIBRARY_PATH = makeLibraryPath [ stdenv.cc.cc.lib zlib ]`.

The loader symlink fixes standalone wheel executables; `LD_LIBRARY_PATH` fixes the C-extension `.so` files dlopened by the store CPython, which never traverse `/lib64` (so the loader symlink alone is insufficient). This is the image-scope analogue of the dev-shell #698 fix, but **ungated**: an all-Nix container has no foreign FHS host binaries to pollute, so the `/etc/NIXOS` ABI gate from #698/#703 never applies. nix-ld was evaluated but is redundant here ‚Äî the mandatory global `LD_LIBRARY_PATH` (needed for the dlopen path nix-ld cannot intercept) already lets a plain glibc-loader symlink cover the standalone-executable case, with a smaller closure.

## Verification

Built `:test-736`, then inside the image:

- `ls -l /lib64/ld-linux-x86-64.so.2` ‚Üí symlink to the Nix glibc loader ‚úì
- `numpy 2.5.0` imports (manylinux wheel, C-ext dlopen) ‚úì
- `pyjson5 OK` (the exact pre-commit `pymarkdown` failure) ‚úì
- PyPI `ruff 0.15.20` runs (standalone manylinux executable via the `/lib64` loader) ‚úì
- `pytest tests/test_image.py::TestManylinuxRuntime` ‚Üí 3 passed

New `tests/test_image.py::TestManylinuxRuntime` asserts the loader exists (unconditional) and that `numpy`/`pyjson5` wheels import under the baked CPython (network-guarded, skips offline).
