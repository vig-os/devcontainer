## Changes Made

- `release.yml`: add candidate-only cross-repo `repository_dispatch` to smoke-test repo, including scoped GitHub App token generation and non-blocking behavior.
- `docs/RELEASE_CYCLE.md`: add manual RC smoke gate phase, update release lifecycle flow, and document `RELEASE_APP` installation requirement on smoke-test repo.
- `CHANGELOG.md`: add unreleased entries for RC smoke dispatch/release gate and related branch changes.
- Workflow and template sync updates across root and workspace template:
  - `codeql.yml`, `scorecard.yml`, `sync-issues.yml`, `ci.yml`, `ci-container.yml`, `release.yml`
  - workspace devcontainer/justfile wiring and manifest/settings sync updates.
- `justfile.gh` / `assets/workspace/.devcontainer/justfile.gh`: remove broken `gh-issues --help` availability guard (the CLI has no `--help` flag, so the check always failed).

- `tests/test_integration.py`: remove `TestGhIssuesSmoke` class (always skipped — no repo context in test workspace; binary availability now covered by image tests).

Top-level workflow permissions grant `pull-requests: write` to all jobs. In this workflow only the `dependency-review` job appears to need PR write, and it already declares job-level `pull-requests: write`. Consider removing the top-level `pull-requests: write` and keeping the narrower job-level permission to follow least-privilege (especially important for reusable workflows via `workflow_call`).

  Not actionable — the top-level `pull-requests: write` is intentional and **required**. When `ci.yml` is called via `workflow_call`, its top-level permissions act as the permission ceiling; job-level permissions cannot exceed it. Removing it would cause `dependency-review` to fail with a permissions error at parse time (before `if` conditions are evaluated). This was already hit and fixed in vig-os/devcontainer-smoke-test#23.
