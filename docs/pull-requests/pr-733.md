# [PR 733](https://github.com/vig-os/devkit/pull/733) fix(devshell): expose python3 and pre-commit on dev-shell PATH (#729)

## Summary

Downstream consumers of the toolchain via the flake-input / direnv path (`vigos.lib.mkProjectShell`, `nix develop`) got a dev-shell with no bare `python` / `python3` and no `pre-commit` on PATH, even though the OCI image provides all three. Python was reachable only via `uv run python`. This is a dev-shell ↔ image parity gap.

Closes #729.

## Root cause

- `devTools` (the toolchain SSoT) deliberately omits a bare interpreter and `pre-commit`. The dev-shell pins `UV_PYTHON` to a Nix-store CPython but puts nothing on PATH.
- `imageTools` adds them separately: a `pythonEnv` (`python314.withPackages`) and `pkgs.pre-commit`. So only the image had parity.
- `mkProjectShell` (hence `devShells.default`) built its package set from `devTools` alone, so the downstream shell lacked both binaries.

## Fix

`mkProjectShell` now appends the pinned store CPython (`python314` — the exact interpreter `UV_PYTHON` already references, reused for DRY) and `pkgs.pre-commit` to the shell's base packages.

Nuance (intentional, explained in the flake comment): these stay **out of** the `devTools` SSoT. The image already provides them via `pythonEnv` + `pre-commit`; adding a bare `python314` to the SSoT would collide with `pythonEnv` in `dockerTools.buildLayeredImage`. The project venv remains uv-managed — exposing `python3` on PATH does not change uv resolution (which goes through `UV_PYTHON`); it just closes the parity gap.

## Tests (TDD)

Added `test_devshell_exposes_python3_and_precommit` (parametrized over `python3` and `pre-commit`) in `tests/test_flake_devshell.py`. It runs under `nix develop --ignore-environment --keep HOME` so it asserts the dev-shell's **own** PATH contribution — a host `python3`/`pre-commit` leaking through the inherited environment can no longer mask the gap (which is exactly how it hid). Written failing first, then fixed.
