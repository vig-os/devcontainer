# [PR 353](https://github.com/vig-os/devcontainer/pull/353) fix(ci): normalize smoke-test workspace ownership after installer

## Summary
- add a post-install write probe in `assets/smoke-test/.github/workflows/repository-dispatch.yml` for repository dispatch deploy runs
- when the probe fails, normalize workspace ownership with `sudo chown -R "$(id -u):$(id -g)" .` before copying `.devcontainer/CHANGELOG.md` to root `CHANGELOG.md`
- document the fix in `CHANGELOG.md` (and synced `assets/workspace/.devcontainer/CHANGELOG.md`) under issue `#352`

The write probe only checks whether a new file can be created in the repo root. That can still succeed even when the existing tracked `CHANGELOG.md` (the file you overwrite a few lines later) is not writable by the runner, so the subsequent `cp ... CHANGELOG.md` can still fail with `Permission denied` without triggering the `chown`. Consider probing writability of the actual target(s) (e.g., `test -w CHANGELOG.md` and/or `test -w .devcontainer`) and running the ownership normalization when those checks fail.

The probe checks `-w` on `.devcontainer/CHANGELOG.md`, but the subsequent `cp` needs the source to be readable (and exist), not writable. If the file ends up non-readable but writable (or is readable-only but root-owned), this check may not reliably predict whether the copy will succeed. Consider probing `-r` (and/or `-f`) for the source, and keep the destination check as `-w`/directory-writable as you already do.
