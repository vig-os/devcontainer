# [PR 460](https://github.com/vig-os/devcontainer/pull/460) fix: finalize release sync and changelog verification

## Description

Reorders release finalize so `sync-issues` runs after the finalize SHA is captured, avoiding a race where downstream build/publish could use a pre-finalize commit. Finalize now fails if `CHANGELOG.md` still contains `## [version] - TBD` after `git reset --hard`. Pre-commit runs `generate-docs` when `CHANGELOG.md` changes so generated README sections stay aligned with the changelog.

The sync-issues wait loop polls `gh run list --limit 1` and assumes the most recent run is the one just triggered. If another sync-issues run is already in progress (schedule/manual) or completes during this window, this can report the wrong status and break early. Capture the run id from the trigger (or query for the newest run created after the trigger time) and poll that specific run via `gh run view`/`gh api` instead.

  Good catch. `gh run list --limit 1` can follow the wrong run if another dispatch or schedule overlaps. This repo's `sync-issues` workflow uses a concurrency group with `cancel-in-progress`, which helps but does not make "newest list row" equivalent to "our dispatch."
