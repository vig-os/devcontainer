# [PR 745](https://github.com/vig-os/devkit/pull/745) fix(image): bake template .venv into Nix image

## Summary

The Nix-built image sets `UV_PROJECT_ENVIRONMENT`/`VIRTUAL_ENV` to `/root/assets/workspace/.venv` but never created that venv (the old Debian image did). The published 0.3.x consumer `post-create.sh` runs `sed -i .../.venv/bin/activate` as its first venv step and aborts under `set -euo pipefail` (`exit 2`) when the file is missing тАФ so git setup, gh auth, pre-commit install, and `just sync` never run. Reproduced on 4/4 non-TypeScript consumers.

## Change

- `flake.nix`: the bootstrap layer (which already creates `/opt/pre-commit-cache`, `/workspace`, `/usr/local/bin`, `/tmp`) now pre-creates the project virtualenv from the baked CPython:
  - `${python}/bin/python3 -m venv "$out/root/assets/workspace/.venv"` тАФ hermetic, network-free, no packages (the consumer's `just sync`/`uv` populates it).
  - Rewrites the build-time `$out` prefix baked into `activate`/`pyvenv.cfg` so `VIRTUAL_ENV` resolves to the runtime path `/root/assets/workspace/.venv` inside the container. The advertised env vars are unchanged (the path is already writable тАФ `assets` is `cp -r`'d, not a store symlink).
- `tests/test_image.py`: new `TestFileStructure::test_template_venv_baked` asserts `/root/assets/workspace/.venv/bin/activate` exists (runs in the CI `image` suite).
- `CHANGELOG.md` `## Unreleased` тЖТ `### Fixed` entry `[#735]`, mirrored into `assets/workspace/.devcontainer/CHANGELOG.md` via `sync_manifest.py`.

## Verification

- Offline-safe: `nix eval --raw .#devcontainerImage.drvPath --accept-flake-config` тЖТ `/nix/store/k5vkv59hx1ln62lvbcwivhyh3pqbg4gy-devcontainer.tar.gz.drv` (flake evaluates).
- Full build + in-container assertion run in the dispatched CI `image` suite (no local build тАФ sandbox has no network).

Closes #735
