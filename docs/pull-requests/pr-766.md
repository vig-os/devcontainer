# [PR 766](https://github.com/vig-os/devkit/pull/766) test(image): parametrized in-container .#devShellTools parity gate

## Summary

Closes #754 (PR #670 review, M1).

The flake's `devShellTools` SSoT was exercised **only** on the dev-shell side (`tests/test_flake_devshell.py`, which is skipped wherever the host lacks `nix`), while `tests/test_image.py` had a hand-curated check covering ~10 of the 27 `devTools`. So adding a tool to `devTools` but not shipping it in the image went uncaught ŌĆö the SSoT was never gated against the running image.

This adds a parametrized presence-on-PATH test in `tests/test_image.py`:

- The tool list is read straight from the flake ŌĆö `nix eval --json .#devShellTools.<system>` ŌĆö at collection time, so it can **never drift** from the SSoT and no tool names are hardcoded (27 cases, matching `devTools`).
- Each tool is asserted to resolve on PATH **inside the running container** via `command -v` (the existing `assert_tool_on_path` idiom).
- This turns the `devShellTools` SSoT into an actual image-side gate: a `devTools` entry missing from the image fails CI.

## Design note

The eval is run **on the host against the checked-out flake** (mirroring the existing `dev_shell_tools`/`current_system` fixtures in `test_flake_devshell.py`), not inside the container. Rationale:

- pytest needs the parametrize ids at collection time (true per-tool parametrization, per the issue AC), which an in-container runtime eval cannot provide.
- The flake is **not** baked into the image, and the image's store lacks the nixpkgs eval-inputs, so an in-container eval would force a network refetch of nixpkgs under `ci.yml`'s 10-min timeout ŌĆö fragile.
- Both image-test CI lanes provision `nix` on the host (`nix-image.yml` installs it; `ci.yml`'s `test-image` provisions via the flake) and check out the flake, so the eval resolves there and the gate never silently no-ops. The per-tool PATH assertion still runs in-container.
