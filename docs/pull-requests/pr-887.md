# [PR 887](https://github.com/vig-os/devkit/pull/887) docs(release): require migrating RC-pinned consumers before RC cleanup

## Description

The 0.4.0 promote's RC-cleanup job deleted all `0.4.0-rc*` git tags and GHCR image versions while two field-validation consumers (EXOMA/hyrr, EXOMA/talys) were still pinned to `DEVCONTAINER_VERSION=0.4.0-rc5`, leaving them unable to pull their devcontainer image until bumped. This PR closes the process gap in the release docs.

Closes #880

## Type of Change

- [x] `docs` -- Documentation only

### Modifiers

- [ ] Breaking change (`!`) -- This change breaks backward compatibility

## Changes Made

- `docs/RELEASE_CYCLE.md`, Phase 5 "Post-Release Cleanup" (manual promote path): new checklist step 2 requiring RC-pinned consumers to be migrated to the final tag before running promote (preferred) or immediately after, naming the concrete mechanism (`DEVCONTAINER_VERSION=...` in the consumer's `.vig-os` file) and the exact job that deletes the RC artifacts (the `cleanup` job, "Cleanup RC artifacts", in `.github/workflows/promote-release.yml`).
- `docs/RELEASE_CYCLE.md`, Release Phases step 6 ("Promote & cleanup"): one-sentence cross-reference to the new Phase 5 step so the cleanup description itself warns about RC-pinned consumers.

The issue's second (optional) acceptance criterion -- having the cleanup job grep known consumer repos' `.vig-os` for `-rc` pins and warn -- is deliberately left out (YAGNI): it would need a cross-org token with read access to private EXOMA repos plus a maintained consumer list, i.e. new infrastructure rather than a small non-blocking addition to the existing job. Can be filed as a follow-up issue if wanted.
