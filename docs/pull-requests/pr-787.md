# [PR 787](https://github.com/vig-os/devkit/pull/787) docs(security): record sops-nix/age + OIDC secrets-management pattern

## Summary

Design-first ADR (issue #780, PR #670 roadmap thread D) recording the
secrets-management **pattern** — not a migration.

- `docs/security/ADR-secrets-management.md` — the decision record:
  - **Two secret classes:** runtime/downstream-consumer secrets → **sops-nix + age**
    (each consumer decrypts with their own key, no per-repo GitHub-secret dance);
    cloud/registry auth → **GitHub OIDC** (the real no-stored-secret lever).
    GitHub App installation tokens (`RELEASE_APP_*`, `COMMIT_APP_*`) stay as-is.
  - **Honest trust-model caveat:** on GitHub-hosted runners SOPS *relocates* the
    root of trust to one bootstrap key (`SOPS_AGE_KEY`) — it does not eliminate
    the GitHub secret. Only OIDC reaches zero stored credentials.
  - **Current landscape cited (file:line):** GHCR-via-`GITHUB_TOKEN` and keyless
    cosign are already correct; per-secret OIDC/sops/keep classification table.
  - **Concrete "now" lever:** honestly, OIDC retires zero *current* secrets today
    (GHCR already tokenless, cosign already keyless, App keys must stay); the win
    is preventive (never add a stored registry/cloud PAT — federate) plus
    `CACHIX_AUTH_TOKEN` as the one realistic future migration target.
- `docs/security/examples/sops-nix/` — a small, clearly-marked **inert** reference
  example (`.sops.yaml`, plaintext `secrets.example.yaml`, `example.nix`, README).
  Placeholder recipients, no real key material, not wired into any live path.
- `CHANGELOG.md` — one `Added` entry (mirrored to the workspace template copy by
  the managed sync-manifest hook).

## Relationship to #786

This is the **plumbing** layer (how secrets are stored/delivered); #786 is the
**agent-behaviour** layer (how an agent must act around a secret). They compose
and do not restate each other — the ADR cross-links #786 and vice-versa.

No workflow is modified. Design only.
